# 运营分析重构设计

## 背景

当前 `/analytics` 已经积累了较多真实数据链路、导入批次切换、版本对比和模块化图表能力，但整体仍然沿用了早期“分类分析页”的结构。它的问题不是没有数据，而是信息架构和页面表达仍然偏向通用图表页：

- 模块差异不够明显，四个业务模块仍然共享过多相同的布局思路
- 页面先给图表，再让用户自己拼结论，不够像“运营分析台”
- 导入质量、推断口径、异常热点与页面主结论之间的关系不够清楚
- 首页缺少一个真正的“运营分析入口”，用户要先知道去哪一页才行

这次重构的目标不是继续在现有页面上小幅补强，而是**直接替换现有 `/analytics` 页面结构**，把它升级成一套按业务模块组织的运营分析系统。

## 目标

本轮重构的目标如下：

1. 左侧导航中的 `运营分析` 成为新的真实入口，取代旧的分类分析认知。
2. `/analytics` 变成运营分析导航台，而不是某个模块页的变体。
3. `/analytics/onboarding`、`/analytics/level`、`/analytics/monetization`、`/analytics/ads` 四个模块页按业务目标重新组织。
4. 每个模块页都遵循“先结论、后主图、再辅助图、再明细”的顺序。
5. 页头必须直接展示数据可信度：
   - 技术通过率
   - 技术异常
   - 业务失败事件
   - 模块覆盖率
   - 是否存在推断统计
6. 数据导入与运营分析形成闭环：
   - 导入页可跳入模块页
   - 运营分析页可跳回对应导入批次预览

## 不在本轮范围内

以下内容不在本轮首批重构范围：

- AI 报告页面的大改版
- 增加新的分析路由层级
- 资源产销、活动、社交等新增运营模块
- 更细粒度的用户分群或 cohort 分析
- 模块三完全独立成页

本轮优先把“导入 -> 运营分析”这条主链路做扎实。

## 用户路径

### 路径 1：从导入进入运营分析

1. 用户在 `/imports` 上传真实日志或生成模拟数据
2. 系统完成清洗、映射、导入摘要、批次保存
3. 用户查看：
   - 技术通过率
   - 技术异常
   - 业务失败事件
   - 模块覆盖率
4. 用户从导入结果区直接点击进入：
   - 新手引导
   - 关卡与局内行为
   - 商业化
   - 广告分析

### 路径 2：从运营分析回查导入批次

1. 用户在任意 `/analytics/...` 模块页查看分析结果
2. 页头直接看到当前批次的数据质量和推断口径
3. 用户点击质量卡
4. 页面跳转到 `/imports?projectId=...&importId=...`
5. 导入页恢复到对应批次的摘要与预览

### 路径 3：从运营分析首页进入模块

1. 用户进入 `/analytics`
2. 先看当前项目与当前批次是否值得分析
3. 再看四个模块入口卡
4. 选择最值得优先处理的模块进入对应模块页

## 新的信息架构

### 路由结构

- `/analytics`
  - 运营分析首页
- `/analytics/onboarding`
  - 新手引导运营分析
- `/analytics/level`
  - 关卡与局内行为运营分析
- `/analytics/monetization`
  - 商业化运营分析
- `/analytics/ads`
  - 广告运营分析

`/analytics/system` 和 `/analytics/custom` 不作为运营分析首批主模块，不再作为主导航优先入口。后续如果保留，也应降级为次级数据类目，而不是一等模块。

### 左侧导航

左侧菜单项保留 `运营分析`，继续指向 `/analytics`。

### 运营分析首页结构

`/analytics` 页面结构如下：

1. 顶部总览区
   - 当前项目
   - 当前导入批次
   - 当前版本 / 对比版本
   - 技术通过率
   - 技术异常
   - 业务失败事件
   - 模块覆盖率

2. 数据可信度提示区
   - 当前是否为显式统计优先
   - 当前是否含推断统计
   - 建议优先阅读的模块

3. 模块入口卡区
   - 新手引导
   - 关卡与局内行为
   - 商业化
   - 广告分析

4. 异常优先入口区
   - 最大流失步骤
   - 失败最集中关卡
   - 最大转化损耗点
   - 最弱广告位

## 模块页统一骨架

四个模块页统一采用以下骨架：

1. 页头
   - 当前项目
   - 当前导入批次
   - 当前版本 / 对比版本
   - 数据来源标签

2. 数据质量卡
   - 技术通过率
   - 技术异常
   - 业务失败事件
   - 模块覆盖率
   - 推断口径提示

3. 关键结论卡
   - 4 张固定指标卡

4. 模块异常信号区
   - 3 张模块专属信号卡

5. 主图区
   - 模块专用主图

6. 辅助图区
   - 2 张模块专用辅助图

7. 明细区
   - 筛选条
   - 结构化明细表

8. 操作与回查区
   - 导出
   - 返回导入预览
   - 切换导入批次
   - 切换版本对比

## 模块级页面设计

### 新手引导 `/analytics/onboarding`

#### 页面目标

帮助团队快速识别：

- 玩家在哪个步骤掉得最多
- 哪个步骤耗时异常
- 版本变化是否影响引导完成率

#### 顶部关键指标

- 首步到达率
- 整体完成率
- 最大流失步骤
- 平均步骤耗时

#### 模块异常信号

- 最大流失步骤
- 耗时异常步骤
- 完成率波动最大的步骤

#### 主图

`步骤漏斗图`

每个步骤展示：

- 到达数
- 完成数
- 流失数
- 完成率
- 平均耗时

最大流失步骤必须在视觉上高亮。

#### 辅助图

- 步骤完成率曲线
- 步骤耗时排行

#### 明细表

字段：

- 步骤名
- 到达数
- 完成数
- 完成率
- 流失数
- 平均耗时
- 说明

### 关卡与局内行为 `/analytics/level`

#### 页面目标

帮助团队快速识别：

- 哪一关卡住了
- 失败集中在哪
- 玩家在关内的行为结构是否异常

#### 顶部关键指标

- 开始数
- 通关率
- 失败率
- 平均重试次数

#### 模块异常信号

- 失败最集中关卡
- 重试最高关卡
- 行为占比异常关卡

#### 主图

`关卡进度漏斗 / 对比图`

每关展示：

- 开始
- 完成
- 失败
- 重试

#### 辅助图

- 失败原因分布
- 重试排行

#### 局内微观心流子区块

并入本页下半区，不单独成页。

展示：

- 各关卡行为占比
- 高频操作排行
- 行为耗时

#### 明细区

分成两段：

1. 关卡明细
   - 关卡
   - 开始
   - 完成
   - 失败
   - 重试
   - 主要失败原因

2. 心流明细
   - 关卡 / 行为
   - 次数
   - 占比
   - 平均耗时
   - 说明

## 数据口径

### 导入质量口径

导入质量必须明确区分两类：

1. 技术导入质量
   - 解析成功率
   - 缺失关键字段
   - 时间格式异常
   - 未识别事件数

2. 业务失败结果
   - `level_fail`
   - `tutorial_step_fail`
   - `result = fail/error`
   - 支付失败
   - 广告点击后未完成

业务失败事件保留进入运营分析，不再压低导入通过率。

### 推断口径

页面必须显式标明当前是否使用推断统计。

推断场景示例：

- 广告日志中没有严格区分 request 与 play
- 商业化日志缺少完整支付阶段，只能按可识别链路统计
- 部分字段来自 `event_value / custom_data` 展开并归一化

### 批次与版本

每个模块页都必须同时保留：

- 当前导入批次切换
- 当前版本
- 对比版本

批次切换和版本切换必须共同生效，不允许互相覆盖上下文。

## 组件边界

### 保留并继续复用

- 导入批次切换组件
- 版本对比组件
- 导出能力
- 导入预览接口

### 重做重点

- `/analytics` 首页
- 模块页页头与信息架构
- 模块专属主图组合
- 模块异常信号区
- 明细表顺序与说明

### 不再作为主渲染核心

原来的通用：

- `main / trend / aux / ranking`

仍可保留为兼容数据层，但**页面不再以这套通用结构作为主信息架构**。页面渲染优先使用模块专用结构。

## 错误处理与空态

### 无真实数据

页面可打开，但必须明确显示：

- 当前为演示态 / fallback
- 当前批次暂无该模块足够数据

### 数据不足但可部分分析

页面继续展示，但必须明确显示：

- 当前为部分推断统计
- 哪些指标是显式统计
- 哪些指标是推断值

### 批次不存在或已删除

- 回落到当前项目最新批次
- 页头给出提示

## 测试要求

需要覆盖以下场景：

1. 导入真实日志后
   - 新手引导页显示真实步骤漏斗
   - 关卡页显示真实关卡漏斗和心流
   - 商业化页显示双漏斗和礼包分布
   - 广告页显示广告位流转

2. 切换导入批次后
   - 页头质量卡同步变化
   - 主图、辅助图、明细同步变化

3. 切换对比版本后
   - 指标卡差值变化
   - 版本差异总览变化

4. 从运营分析回导入预览
   - 点页头质量卡可跳到对应 `projectId + importId`

5. 无真实数据时
   - 页面仍能打开
   - 明确显示演示态

## 实施顺序

按以下顺序替换：

1. 重做 `/analytics` 运营分析首页
2. 重做 `/analytics/onboarding`
3. 重做 `/analytics/level`
4. 重做 `/analytics/monetization`
5. 重做 `/analytics/ads`

原因：

- 首页先定信息架构和模块入口
- 新手引导与关卡最接近现有真实数据
- 商业化与广告在口径上更依赖推断，放后面更稳

## 风险与注意点

1. 当前代码库里已经有较多 analytics 旧结构，重构时容易继续沿用旧卡片思路。
   - 需要主动以“模块页重做”为目标，而不是“旧页增强”。

2. 商业化和广告口径在首版会有推断成分。
   - 必须在 UI 上显式说明，不能伪装成完全精确。

3. 首页和模块页的视觉语言必须统一。
   - 否则会继续让用户觉得只是旧页改文案。

4. 重构中要保住现有的批次切换、版本切换、导出能力。
   - 不要为了重做页面而把已可用链路破坏掉。

## 结论

这次重构的目标不是继续把 `/analytics` 修得更像“好一点的旧页面”，而是把它真正替换成：

- 一个运营分析首页
- 四个模块专属分析页
- 一套清楚的数据质量与批次回查闭环

只有这样，页面看起来和使用起来才会真正和原来的“分类分析页”不一样。
